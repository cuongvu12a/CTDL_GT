## 1. Kubernetes là gì?

### 1.1 Định nghĩa ngắn gọn

**Kubernetes (K8s)** là một nền tảng **điều phối container (container orchestration)** mã nguồn mở, được Google phát triển và hiện do CNCF (Cloud Native Computing Foundation) duy trì.

Nói đơn giản: **Kubernetes là "hệ điều hành" cho các ứng dụng container hóa**. Giống như Linux quản lý process trên một máy, Kubernetes quản lý container trên một cụm máy (cluster).

### 1.2 Bài toán mà Kubernetes giải quyết

Trước khi có Kubernetes, việc vận hành ứng dụng container hóa trên production gặp nhiều vấn đề:

```
Không có K8s:
┌─────────────────────────────────────────────────────────────┐
│  - Ứng dụng chết → phải SSH vào server restart thủ công    │
│  - Traffic tăng  → phải thêm máy, cấu hình LB thủ công    │
│  - Deploy phiên bản mới → downtime, rollback phức tạp      │
│  - 10 microservices → 10 server riêng, lãng phí tài nguyên │
│  - Config khác nhau giữa dev/staging/prod → lỗi liên tục   │
└─────────────────────────────────────────────────────────────┘

Có K8s:
┌─────────────────────────────────────────────────────────────┐
│  - Ứng dụng chết → K8s tự restart trong vài giây           │
│  - Traffic tăng  → K8s tự scale thêm container             │
│  - Deploy mới    → rolling update, zero-downtime            │
│  - 10 microservices → K8s tự xếp tối ưu lên các node       │
│  - Config → khai báo 1 lần trong YAML, apply mọi nơi       │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 Kiến trúc tổng quan (chỉ cần nhớ 2 phần)

```
┌──────────────────────────────────────────────────────────┐
│                    KUBERNETES CLUSTER                     │
│                                                          │
│  ┌──────────────────────────────────────────────┐        │
│  │           CONTROL PLANE (Master)              │        │
│  │                                               │        │
│  │  ┌───────────┐  ┌────────────┐  ┌─────────┐ │        │
│  │  │ API Server│  │ Scheduler  │  │  etcd   │ │        │
│  │  │ (Cổng     │  │ (Xếp Pod   │  │ (Lưu   │ │        │
│  │  │  chính)   │  │  vào Node)  │  │  state) │ │        │
│  │  └───────────┘  └────────────┘  └─────────┘ │        │
│  │  ┌──────────────────────────┐                │        │
│  │  │  Controller Manager      │                │        │
│  │  │  (Đảm bảo desired state) │                │        │
│  │  └──────────────────────────┘                │        │
│  └──────────────────────────────────────────────┘        │
│                                                          │
│  ┌──────────────────┐  ┌──────────────────┐              │
│  │   WORKER NODE 1   │  │   WORKER NODE 2   │            │
│  │                    │  │                    │            │
│  │  ┌──────┐┌──────┐ │  │  ┌──────┐┌──────┐ │            │
│  │  │ Pod  ││ Pod  │ │  │  │ Pod  ││ Pod  │ │            │
│  │  │(app) ││(api) │ │  │  │(app) ││ (db) │ │            │
│  │  └──────┘└──────┘ │  │  └──────┘└──────┘ │            │

```

**Chỉ cần nhớ:**
- **Control Plane** = Bộ não, ra quyết định ("cần 3 Pod, Pod này chết thì tạo lại")
- **Worker Node** = Máy thực thi, chạy container thật sự
- **Pod** = Đơn vị nhỏ nhất, chứa 1 hoặc nhiều container
- **kubectl** = CLI để nói chuyện với Control Plane

### 1.4 Khái niệm "Desired State" — Triết lý cốt lõi

Kubernetes hoạt động theo mô hình **khai báo (declarative)**, không phải mệnh lệnh (imperative):

```
Imperative (truyền thống):       Declarative (K8s):
"Hãy khởi động 3 server"        "Tôi muốn luôn có 3 instance chạy"
"Hãy restart server #2"         → K8s tự restart khi cần
"Hãy thêm 2 server nữa"        → K8s tự scale khi cần
```

Bạn **khai báo trạng thái mong muốn** (desired state) trong file YAML, Kubernetes **liên tục đảm bảo** hệ thống khớp với trạng thái đó. Nếu lệch (Pod chết, node hỏng), K8s tự sửa.

## 2. Kubernetes đóng góp gì cho hệ thống Client–Server?

### 2.1 Luồng request truyền thống vs. có Kubernetes

```
TRUYỀN THỐNG:
Client → DNS → Load Balancer (Nginx/HAProxy) → Server (1 máy vật lý)
                                                 ↓
                                            Nếu chết → downtime
                                            Nếu quá tải → thủ công thêm máy


CÓ KUBERNETES:
Client → DNS → Ingress Controller → Service → Pod(s)
                   │                    │         │
                   │                    │         ├─ Pod 1 (container app)
                   │                    │         ├─ Pod 2 (container app)
                   │                    │         └─ Pod 3 (container app)
                   │                    │
                   │                    └─ Load balancing tự động
                   │                       Health check tự động
                   │                       Loại Pod lỗi khỏi pool
                   │
                   └─ Routing theo domain/path
                      TLS termination
                      Rate limiting
```

### 2.2 Các đóng góp cụ thể cho hệ thống Client–Server

| Khía cạnh | Không có K8s | Có K8s |
|---|---|---|
| **High Availability** | Cấu hình thủ công failover, keepalived, HAProxy | K8s tự restart Pod, reschedule lên node khác khi node chết |
| **Scaling** | Thêm máy thủ công, cập nhật LB config | `kubectl scale` hoặc HPA tự scale theo CPU/memory/custom metrics |
| **Load Balancing** | Cài và cấu hình Nginx/HAProxy riêng | Service tự động distribute traffic đến các Pod healthy |
| **Zero-downtime Deploy** | Blue-green phức tạp, cần setup riêng | Rolling update mặc định, rollback 1 lệnh |
| **Service Discovery** | Hardcode IP, hoặc dùng Consul/etcd riêng | Tự động qua DNS nội bộ: `my-service.my-namespace.svc.cluster.local` |
| **Config Management** | File .env trên mỗi server, dễ lệch | ConfigMap + Secret, quản lý tập trung, version controlled |
| **Resource Optimization** | Mỗi service 1 VM, lãng phí 60-70% | K8s bin-pack nhiều Pod vào ít node, tận dụng ~70-80% tài nguyên |
| **Self-Healing** | Cần supervisor, monit, cron job | Tự restart container lỗi, tự thay thế Pod trên node hỏng |
| **Networking** | Cấu hình iptables, firewall thủ công | Network Policy khai báo, mỗi Pod có IP riêng |

### 2.3 Ví dụ minh hoạ: Một hệ thống e-commerce

```
                          ┌──── KUBERNETES CLUSTER ─────────────────┐
                          │                                          │
Client ──→ Ingress ──────→│  ┌─────────┐    ┌──────────┐            │
(Browser)  (nginx-ingress)│  │ Web App  │───→│ Order    │            │
                          │  │ (3 pods) │    │ Service  │            │
                          │  └─────────┘    │ (2 pods) │            │
                          │                  └────┬─────┘            │
                          │                       │                  │
                          │                  ┌────▼─────┐            │
                          │                  │ Payment  │            │
                          │                  │ Service  │            │
                          │                  │ (2 pods) │            │
                          │                  └────┬─────┘            │
                          │                       │                  │
                          │                  ┌────▼─────┐            │
                          │                  │PostgreSQL│            │
                          │                  │(StatefulSet)          │
                          │                  └──────────┘            │
                          └──────────────────────────────────────────┘

Khi traffic tăng (Flash Sale):
  - HPA tự scale Web App: 3 → 10 pods
  - HPA tự scale Order Service: 2 → 5 pods
  - Service tự load balance traffic mới vào pods mới
  - Khi traffic giảm → tự scale về lại
  → Không cần can thiệp thủ công
```

## 3. Kubernetes được sử dụng như thế nào trong thực tế?

### 3.1 Các use case phổ biến

#### a) Microservices Orchestration (phổ biến nhất)
- Mỗi microservice là 1 Deployment + Service
- K8s quản lý inter-service communication qua DNS nội bộ
- Scale từng service độc lập theo nhu cầu
- **Ví dụ**: Spotify, Airbnb, Pinterest đều chạy hàng ngàn microservice trên K8s

#### b) CI/CD & Automated Deployment
- Pipeline: Push code → Build Docker image → Push registry → K8s rolling update
- Rollback tức thì nếu phiên bản mới lỗi: `kubectl rollout undo`
- GitOps (ArgoCD/FluxCD): Git repo là single source of truth cho cluster state
- **Ví dụ**: Team push code → 5 phút sau live trên production, zero-downtime

#### c) Multi-environment Management
- Namespace tách biệt: `dev`, `staging`, `production` trên cùng cluster
- Hoặc cluster riêng cho mỗi môi trường
- Cùng YAML manifests, chỉ khác values (qua Helm/Kustomize)
- **Ví dụ**: Developer tạo preview environment cho mỗi PR

#### d) Batch Processing & Job Scheduling
- CronJob cho các tác vụ định kỳ (report, ETL, cleanup)
- Job cho one-time task (data migration, batch processing)
- **Ví dụ**: Chạy data pipeline hàng đêm, tự tạo Pod xử lý xong tự hủy

#### e) Edge Computing & Hybrid Cloud
- K8s chạy được ở mọi nơi: on-premise, AWS, GCP, Azure, edge device
- Một bộ manifest, deploy ở bất kỳ đâu
- **Ví dụ**: Chạy K8s trên cả data center nội bộ và cloud, failover giữa 2 nơi

### 3.2 Khi nào KHÔNG cần Kubernetes?

Quan trọng không kém — **K8s không phải silver bullet**:

| Trường hợp | Giải pháp thay thế tốt hơn |
|---|---|
| Ứng dụng monolith đơn giản, 1-2 server | Docker Compose, hoặc VM thông thường |
| Team nhỏ (1-3 người), dự án nhỏ | PaaS: Railway, Render, Fly.io, Heroku |
| Serverless workload | AWS Lambda, Cloud Functions |
| Chưa dùng Docker | Containerize trước, K8s sau |
| Startup giai đoạn đầu, cần ship nhanh | Managed PaaS, đừng tốn thời gian vào infra |

**Quy tắc ngón tay cái**: Nếu bạn có < 5 services và team < 5 người, hãy cân nhắc thật kỹ trước khi dùng K8s. Chi phí vận hành (complexity, learning curve, debugging) có thể lớn hơn lợi ích.

### 3.3 Kubernetes trong dự án thực tế — Flow điển hình

```
Developer workflow:
 1. Code → Push Git
 2. CI pipeline trigger:
    - Run tests
    - Build Docker image
    - Push image lên Container Registry (ECR/GCR/DockerHub)
 3. CD pipeline:
    - Update image tag trong K8s manifest/Helm chart
    - kubectl apply hoặc helm upgrade
    - K8s rolling update: tạo Pod mới → health check OK → terminate Pod cũ
 4. Monitoring:
    - Prometheus thu thập metrics
    - Grafana dashboard hiển thị
    - Alert nếu error rate tăng → auto rollback hoặc notify team
```

## 4. Đánh giá lộ trình học hiện tại

### 4.1 Điểm mạnh

| # | Điểm mạnh | Nhận xét |
|---|---|---|
| 1 | **Phương pháp DO → SEE → ASK → READ → DO** | Xuất sắc. Đây là cách học hiệu quả nhất cho công cụ operations |
| 2 | **Chia 4 Phase rõ ràng** | Progression hợp lý: nền tảng → thực hành → maintain → nâng cao |
| 3 | **Phase 4 dạng "học khi cần"** | Đúng triết lý — không cần học trước những gì chưa gặp |
| 4 | **Có bài tập thực hành cụ thể** | Rất tốt, không chỉ lý thuyết |
| 5 | **Có bảng troubleshooting** | Thực tế, đây là thứ dùng hàng ngày |

### 4.2 Điểm cần điều chỉnh

| # | Vấn đề | Chi tiết |
|---|---|---|
| 1 | **Phase 1 quá chi tiết cho "nền tảng"** | 1.1 → 1.6 giải thích rất kỹ từng bước nhỏ (15 phút, 10 phút, 5 phút). Với đối tượng Senior BE/Tech Lead, nhiều khái niệm (container, deployment, scaling) đã quen — không cần cầm tay chỉ việc đến mức này |
| 2 | **Thiếu mental model tổng quan** | Nhảy thẳng vào `minikube start` mà không có bức tranh toàn cảnh "K8s giải quyết bài toán gì". Senior engineer cần hiểu **WHY** trước khi **HOW** |
| 3 | **Phase 2 và Phase 3 có thể gộp và rút gọn** | ConfigMap, Secret, Namespace ở Phase 2 thực tế sẽ gặp ngay khi đọc manifest công ty ở Phase 3. Học tách rời rồi lại áp dụng lại = lặp |
| 4 | **Timeline 6-9 tuần hơi dài cho mục tiêu "high-level + maintain"** | Với Senior BE đã quen Docker, 3-4 tuần tập trung là đủ cho mục tiêu ban đầu |
| 5 | **Thiếu phần mapping K8s concepts ↔ công ty** | Nên có bước sớm để đọc sơ setup công ty, biết cần tập trung vào đâu, thay vì học generic rồi mới áp dụng |
| 6 | **Phần Helm quá sơ sài** | Chỉ liệt kê 4 lệnh. Trong thực tế maintain, Helm chart values, template logic là thứ phải đụng hàng ngày |

### 4.3 Tổng kết đánh giá

Lộ trình hiện tại **chất lượng tốt** và **đi đúng hướng**. Phương pháp "thực hành trước — lý thuyết sau" là chính xác. Tuy nhiên, với mục tiêu cụ thể (hiểu tổng quan + deploy local + maintain dự án công ty), có thể **rút gọn và tập trung hơn** bằng cách:

- Bỏ phần hướng dẫn quá chi tiết (mang tính tutorial) — Senior engineer tự làm được
- Gộp Phase 2 + Phase 3 thành một luồng thực hành liền mạch
- Thêm bước "reconnaissance" sớm — xem setup công ty trước để biết cần học gì
- Rút timeline từ 6-9 tuần xuống 3-4 tuần

### Phase 0 — Bức tranh lớn + Reconnaissance (2 ngày)

> **Mục tiêu**: Hiểu K8s giải quyết bài toán gì, và biết setup K8s công ty mình đang dùng những gì.

#### Ngày 1: Hiểu "Why Kubernetes"

**Lý thuyết (2-3 giờ):**
- [ ] Đọc Section 1, 2, 3 của tài liệu này (K8s là gì, đóng góp gì, use case)
- [ ] Xem video "Kubernetes Explained" của TechWorld with Nana (~30 phút)
- [ ] Đọc [Kubernetes Concepts Overview](https://kubernetes.io/docs/concepts/overview/) (chỉ phần Overview, không đi sâu)

**Kết quả cần đạt:** Trả lời được các câu hỏi:
- K8s giải quyết bài toán gì mà Docker Compose không làm được?
- Pod, Deployment, Service, Ingress — mỗi thứ đóng vai trò gì trong luồng request?
- Desired state nghĩa là gì?

#### Ngày 2: Reconnaissance dự án công ty

**Khảo sát (2-3 giờ):**
- [ ] Clone/mở repo chứa K8s config của công ty
- [ ] Liệt kê xem công ty đang dùng những resource nào (Deployment, Service, Ingress, ConfigMap, Secret, HPA, PV, ...)
- [ ] Xác định công ty dùng Helm hay plain YAML
- [ ] Xem CI/CD pipeline — image được build và deploy như thế nào
- [ ] Ghi chú lại những khái niệm/resource chưa hiểu → đây là **danh sách học tập trung**

**Output:**
```
Checklist reconnaissance:
□ Repo K8s config nằm ở đâu?
□ Dùng Helm chart hay plain YAML manifests?
□ Có bao nhiêu service/deployment?
□ Ingress config ra sao? Domain nào?
□ Secret được quản lý thế nào? (K8s Secret, External Secrets, Vault?)
□ CI/CD tool gì? (GitHub Actions, GitLab CI, Jenkins?)
□ Cluster chạy ở đâu? (EKS, GKE, AKS, on-premise?)
□ Monitoring/Logging dùng gì? (Prometheus, Grafana, ELK, Datadog?)
```

> **Tại sao Phase 0 quan trọng?** Vì nó giúp bạn biết **cần tập trung học gì** thay vì học tràn lan. Nếu công ty không dùng StatefulSet thì không cần học. Nếu công ty dùng Helm nhiều thì phải ưu tiên Helm.

### Phase 1 — Hands-on cơ bản (1 tuần)

> **Mục tiêu**: Tự tay deploy ứng dụng lên minikube, quen kubectl và viết YAML.
> **Phương pháp**: Vẫn giữ DO → SEE → ASK → READ → DO.

#### 1.1 Setup & Core Concepts (Ngày 1-2)

```bash
# Cài đặt
brew install minikube kubectl

# Khởi tạo cluster
minikube start

# Khám phá cluster
kubectl get nodes
kubectl get pods --all-namespaces
kubectl cluster-info
```

**Thực hành nhanh — deploy + phá + sửa:**

```bash
# Deploy nginx
kubectl create deployment my-web --image=nginx

# Xem pod
kubectl get pods

# Scale lên 3
kubectl scale deployment my-web --replicas=3

# Xóa 1 pod, xem self-healing
kubectl delete pod <tên-pod>
kubectl get pods    # Pod mới tự xuất hiện

# Expose ra ngoài
kubectl expose deployment my-web --type=NodePort --port=80
minikube service my-web --url   # Mở browser kiểm tra

# Xem toàn bộ resource
kubectl get all

# Dọn dẹp
kubectl delete deployment my-web
kubectl delete service my-web
```

**Sau khi thực hành xong**, đọc docs để hiểu sâu hơn:
- [Pods](https://kubernetes.io/docs/concepts/workloads/pods/)
- [Deployments](https://kubernetes.io/docs/concepts/workloads/controllers/deployment/)
- [Services](https://kubernetes.io/docs/concepts/services-networking/service/)

#### 1.2 YAML Manifests & kubectl nâng cao (Ngày 3-4)

**Viết manifest đầu tiên:**

```yaml
# my-app.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: nginx:1.25
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: "50m"
              memory: "64Mi"
            limits:
              cpu: "200m"
              memory: "128Mi"
---
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 80
  type: NodePort
```

```bash
# Apply manifest
kubectl apply -f my-app.yaml

# Kiểm tra

# Sửa manifest (đổi replicas: 3) → apply lại
kubectl apply -f my-app.yaml

# Xem history
kubectl rollout history deployment/my-app

# Xem logs
kubectl logs -l app=my-app

```

**kubectl cần thành thạo:**


#### 2.4 Health Checks & Resource Management (Ngày 10)

```yaml
# Liveness: Pod có đang sống không? Fail → K8s restart pod
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10

# Readiness: Pod có sẵn sàng nhận traffic không? Fail → loại khỏi Service
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
```

```yaml
# Resources: Giới hạn CPU/Memory cho container
resources:
  requests:        # Minimum cần để schedule
    cpu: "100m"    # 0.1 CPU core
    memory: "128Mi"
  limits:          # Maximum được phép dùng
    cpu: "500m"    # 0.5 CPU core
    memory: "512Mi" # Vượt memory limit → OOMKilled
```

#### 2.5 Troubleshooting — Kỹ năng quan trọng nhất (Ngày 11-12)

**Bảng tra cứu nhanh:**

| Trạng thái Pod | Nguyên nhân phổ biến | Cách debug |
|---|---|---|
| `CrashLoopBackOff` | App crash liên tục (lỗi code, thiếu env, sai config) | `kubectl logs <pod> --previous` |
| `Pending` | Không đủ resource trên cluster | `kubectl describe pod <pod>` → xem Events |
| `ImagePullBackOff` | Sai image name, tag không tồn tại, thiếu credentials | Kiểm tra image name, `imagePullSecrets` |
| `OOMKilled` | Container dùng memory vượt limit | Tăng limit hoặc optimize app |
| `CreateContainerError` | Thiếu ConfigMap, Secret, hoặc Volume | `kubectl describe pod <pod>` → xem Events |
| Service không kết nối | Selector không match, port sai | `kubectl describe svc`, kiểm tra labels match |
| Ingress không hoạt động | Ingress controller chưa cài, annotation sai | `kubectl describe ingress`, kiểm tra logs ingress controller |

**Quy trình debug chuẩn:**

```bash
# Bước 1: Xem trạng thái tổng quan
kubectl get pods -n <namespace>

# Bước 2: Xem chi tiết pod (Events là phần quan trọng nhất)
kubectl describe pod <pod> -n <namespace>

# Bước 3: Xem logs
kubectl logs <pod> -n <namespace>
kubectl logs <pod> -n <namespace> --previous  # Nếu pod đã restart

# Bước 4: Kiểm tra service connectivity
kubectl get endpoints <service>  # Xem service có target pod nào không

# Bước 5: Test từ trong cluster
kubectl run debug --rm -it --image=busybox -- sh
wget -qO- http://<service>.<namespace>.svc.cluster.local
```
