# Kubernetes — Tổng quan & Lộ trình Học tập

> **Đối tượng:** Solution Architect, Senior Backend Engineers, Tech Leads, Product Managers
> **Mục tiêu:** Nắm vững Kubernetes ở mức high-level, triển khai local, và maintain dự án công ty.

---

## 1. Kubernetes là gì?

### 1.1 Định nghĩa

Kubernetes (viết tắt: **K8s**) là một nền tảng **container orchestration** mã nguồn mở, ban đầu được Google phát triển, hiện do **CNCF** (Cloud Native Computing Foundation) duy trì.

Nói đơn giản: Kubernetes **tự động hóa** việc triển khai, mở rộng và vận hành các ứng dụng chạy trong container (Docker, containerd, …).

### 1.2 Vị trí của K8s trong kiến trúc hệ thống

```
┌─────────────────────────────────────────────────────────────┐
│                     End Users / Clients                     │
└──────────────────────────┬──────────────────────────────────┘
                           │  HTTPS
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              Load Balancer / Ingress Controller             │
└──────────────────────────┬──────────────────────────────────┘
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
   ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
   │   Pod (A)   │ │   Pod (B)   │ │   Pod (C)   │  ← Replicas
   │  Container  │ │  Container  │ │  Container  │
   └──────┬──────┘ └──────┬──────┘ └──────┬──────┘
          │               │               │
          └───────────────┼───────────────┘
                          ▼
            ┌──────────────────────────┐
            │   Database / Services    │
            │  (StatefulSet, PV/PVC)   │
            └──────────────────────────┘

  ← Tất cả được quản lý bởi Kubernetes Cluster →
```

### 1.3 Tại sao không chỉ dùng Docker?

| Vấn đề khi chỉ dùng Docker               | Kubernetes giải quyết bằng cách     |
| ------------------------------------------ | ------------------------------------ |
| Container crash → downtime                 | Tự restart, tự heal (self-healing)   |
| Scale thủ công                             | Auto-scaling (HPA)                   |
| Deploy mới → phải dừng cũ                 | Rolling update, zero-downtime deploy |
| Quản lý config/secret phân tán            | ConfigMap & Secret tập trung         |
| Networking giữa các container phức tạp    | Service discovery tự động            |
| Không có load balancing built-in           | Service + Ingress                    |

## 2. Kubernetes đóng góp gì cho hệ thống Client–Server?

### 2.1 Từ góc nhìn Solution Architect

Kubernetes không thay đổi **mô hình client–server**, mà nâng cấp **lớp vận hành phía server** với các giá trị cốt lõi sau:

#### a) High Availability (Tính sẵn sàng cao)
- Nhiều replica của cùng một service chạy song song.
- Khi một Pod chết, K8s tự tạo Pod mới trong vài giây.
- Không còn tình trạng "server chết → cả hệ thống sập".

#### b) Horizontal Scaling (Mở rộng ngang)
- `HorizontalPodAutoscaler` tự tăng/giảm số replica dựa trên CPU, memory hoặc custom metrics.
- Phù hợp với traffic không ổn định (flash sale, giờ cao điểm).

#### c) Service Discovery & Load Balancing
- Mỗi Deployment có một **Service** (ClusterIP, NodePort, LoadBalancer).
- Các service gọi nhau qua DNS nội bộ: `http://payment-service.default.svc.cluster.local`.
- Không cần hardcode IP, không cần Consul/Eureka riêng biệt (vẫn dùng được nếu muốn).

#### d) Declarative Configuration (Cấu hình khai báo)
- Toàn bộ hạ tầng mô tả bằng YAML → version control với Git.
- **GitOps**: push code → CI build image → CD apply YAML → K8s tự deploy.

#### e) Zero-Downtime Deployment
- **Rolling Update**: thay Pod mới dần dần, chỉ gỡ Pod cũ khi Pod mới healthy.
- **Canary / Blue-Green**: kiểm soát traffic khi release tính năng mới.

#### f) Secret & Config Management
- Tách biệt config khỏi image: thay đổi config không cần rebuild.
- Secret được mã hóa, mount vào Pod dưới dạng file hoặc biến môi trường.

### 2.2 Tóm tắt giá trị kinh doanh

| Giá trị kỹ thuật          | Giá trị kinh doanh                              |
| -------------------------- | ------------------------------------------------ |
| Self-healing               | Giảm downtime, giảm chi phí on-call             |
| Auto-scaling               | Tối ưu chi phí cloud, chịu tải tốt hơn         |
| Rolling update             | Release thường xuyên hơn, ít rủi ro hơn         |
| Declarative + GitOps       | Audit trail, reproduce môi trường dễ dàng        |
| Multi-environment          | Dev/Staging/Prod nhất quán                       |

## 3. Các use case phổ biến của Kubernetes

### 3.1 Microservices Architecture
- Mỗi microservice = 1 Deployment + 1 Service.
- K8s quản lý networking, scaling, health check cho từng service độc lập.

### 3.2 CI/CD Pipeline
- Build image → Push registry → K8s apply manifest → Tự động deploy.
- Công cụ phổ biến: ArgoCD, FluxCD, GitHub Actions, GitLab CI.

### 3.3 Batch Processing & Cron Jobs
- `Job` và `CronJob` cho tác vụ xử lý hàng loạt (ETL, report, data migration).

### 3.4 Machine Learning Workloads
- Kubeflow chạy trên K8s để train/serve ML models.
- GPU scheduling với Node selectors.

### 3.5 Multi-tenant SaaS
- **Namespace** cách ly tài nguyên giữa các tenant.
- **ResourceQuota** giới hạn CPU/memory cho từng namespace.

### 3.6 Edge / IoT
- K3s (lightweight K8s) chạy trên thiết bị edge.
- Quản lý tập trung từ cloud cluster.

## 4. Phân tích lộ trình học tập thông thường

### 4.1 Vấn đề của lộ trình quá chi tiết

Hầu hết các roadmap Kubernetes phổ biến (ví dụ: roadmap.sh/kubernetes) đi sâu vào:

- Cài đặt cluster production từ đầu (kubeadm, etcd, PKI)
- Network plugin chi tiết (Calico, Cilium, Flannel internals)
- Storage backend (Ceph, Rook, CSI driver)
- RBAC phức tạp, OPA/Gatekeeper
- Custom Resource Definitions (CRD) & Operators
- Service Mesh (Istio, Linkerd)
- eBPF, kernel-level networking

**Những kiến thức này quan trọng nhưng KHÔNG cần thiết ở giai đoạn đầu** khi mục tiêu là:
1. Hiểu K8s hoạt động ra sao.
2. Chạy được ứng dụng trên local.
3. Maintain dự án đang có.

### 4.2 Rủi ro khi học quá sâu quá sớm
- **Overwhelm**: quá nhiều concept cùng lúc → bỏ cuộc.
- **Thiếu context**: học Network Policy khi chưa hiểu Pod là gì → không thấy giá trị.
- **Chậm áp dụng**: mất vài tháng mới bắt tay vào dự án thực tế.

## 5. Lộ trình học tập đề xuất

> **Nguyên tắc:** Học vừa đủ để **làm được việc** → Dùng thực tế → Học sâu khi gặp vấn đề cụ thể.

### Phase 1: Nền tảng — Hiểu K8s là gì (1–2 tuần)

**Mục tiêu:** Trả lời được "K8s giải quyết vấn đề gì?" và hiểu các khái niệm cốt lõi.

| Chủ đề                  | Nội dung cần nắm                                                  | Mức độ    |
| ------------------------ | ------------------------------------------------------------------ | --------- |
| Container cơ bản        | Docker image, container, Dockerfile, docker-compose                | Bắt buộc  |
| K8s Architecture         | Control Plane (API Server, Scheduler, Controller Manager, etcd)    | Hiểu sơ đồ|
|                          | Worker Node (kubelet, kube-proxy, container runtime)               | Hiểu sơ đồ|
| Core Objects             | Pod, Deployment, Service, Namespace                                | Bắt buộc  |
| YAML Manifest            | Cấu trúc YAML cơ bản, cách khai báo Deployment + Service          | Bắt buộc  |
| kubectl cơ bản          | `get`, `describe`, `logs`, `apply`, `delete`, `exec`               | Bắt buộc  |

**Tài liệu đề xuất:**
- [Kubernetes Official — Concepts](https://kubernetes.io/docs/concepts/)
- Video: "Kubernetes Explained in 15 Minutes" (TechWorld with Nana)
- Interactive: [Play with Kubernetes](https://labs.play-with-k8s.com/)

---

### Phase 2: Thực hành Local — Deploy ứng dụng (1–2 tuần)

**Mục tiêu:** Tự tay deploy ứng dụng trên máy local, hiểu luồng từ YAML → running Pod.

| Chủ đề                  | Nội dung cần nắm                                                  | Mức độ    |
| ------------------------ | ------------------------------------------------------------------ | --------- |
| Local Cluster            | Cài **minikube** hoặc **Docker Desktop K8s** hoặc **kind**         | Bắt buộc  |
| Hands-on Deploy          | Deploy 1 app (ví dụ: nginx, hoặc app BE của bạn)                  | Bắt buộc  |
| Service Types            | ClusterIP, NodePort, LoadBalancer — khi nào dùng cái nào           | Bắt buộc  |
| ConfigMap & Secret       | Inject config/secret vào Pod                                       | Bắt buộc  |
| Port-forward & Logs      | `kubectl port-forward`, `kubectl logs -f`                          | Bắt buộc  |
| Namespace                | Tạo namespace, deploy vào namespace khác nhau                      | Nên biết  |

**Bài tập thực hành đề xuất:**
```bash
# 1. Tạo cluster local
minikube start

# 2. Deploy nginx
kubectl create deployment nginx --image=nginx
kubectl expose deployment nginx --port=80 --type=NodePort
minikube service nginx

# 3. Deploy app của bạn
kubectl apply -f my-app-deployment.yaml
kubectl apply -f my-app-service.yaml
kubectl get pods -w          # watch pods starting
kubectl logs -f <pod-name>   # xem logs
```

---

### Phase 3: Maintain dự án công ty (2–3 tuần)

**Mục tiêu:** Đọc hiểu và thao tác được với K8s manifests của dự án hiện tại.

| Chủ đề                      | Nội dung cần nắm                                                   | Mức độ    |
| ---------------------------- | ------------------------------------------------------------------- | --------- |
| Đọc hiểu manifests          | Hiểu các YAML file trong repo dự án (Deployment, Service, Ingress) | Bắt buộc  |
| Ingress                      | Ingress Controller (nginx-ingress), routing rules, TLS              | Bắt buộc  |
| Health Checks                | `livenessProbe`, `readinessProbe`, `startupProbe`                   | Bắt buộc  |
| Resource Limits              | `requests` vs `limits` cho CPU & Memory                             | Bắt buộc  |
| Rolling Update               | `strategy.rollingUpdate`, `maxSurge`, `maxUnavailable`              | Bắt buộc  |
| Helm cơ bản                 | Helm chart là gì, `helm install`, `helm upgrade`, `values.yaml`     | Nên biết  |
| Debugging                    | Pod ở trạng thái CrashLoopBackOff, ImagePullBackOff, Pending       | Bắt buộc  |
| Kết nối với CI/CD           | Hiểu pipeline deploy lên K8s (không cần tự build)                  | Nên biết  |

**Checklist trước khi maintain dự án:**
- [ ] Biết cluster dự án chạy ở đâu (EKS, GKE, AKS, on-prem?)
- [ ] Có quyền `kubectl` truy cập cluster
- [ ] Đọc được toàn bộ YAML manifests / Helm charts trong repo
- [ ] Biết cách xem logs, describe pod, exec vào container
- [ ] Hiểu pipeline CI/CD deploy lên K8s như thế nào
- [ ] Biết quy trình rollback khi deploy lỗi

### Phase 4: Nâng cao — Học khi cần (Sau Phase 3)

> **Không học ngay.** Chỉ học khi gặp vấn đề thực tế hoặc muốn optimize hệ thống.

| Chủ đề                          | Khi nào cần học                                           |
| -------------------------------- | --------------------------------------------------------- |
| HPA (Horizontal Pod Autoscaler) | Khi cần auto-scale theo traffic                           |
| PersistentVolume / PVC           | Khi app cần lưu trữ dữ liệu (database, file upload)     |
| StatefulSet                      | Khi deploy database (Postgres, Redis) trên K8s            |
| Network Policy                   | Khi cần kiểm soát traffic giữa các service               |
| RBAC                             | Khi cần quản lý quyền truy cập cho nhiều team             |
| Service Mesh (Istio/Linkerd)     | Khi hệ thống microservices phức tạp, cần observability    |
| Custom Resource / Operators      | Khi cần mở rộng K8s cho nhu cầu đặc biệt                |
| Cluster Setup (kubeadm)          | Khi cần tự dựng cluster production on-prem                |
| GitOps (ArgoCD, FluxCD)          | Khi muốn automate deployment flow hoàn toàn               |
| Monitoring (Prometheus/Grafana)  | Khi cần giám sát chi tiết hệ thống                       |

## 8. Tóm tắt cho Product Manager & Tech Lead

> **Kubernetes giúp gì cho dự án?**

1. **Ổn định hơn**: App tự khởi động lại khi lỗi, không cần can thiệp thủ công.
2. **Scale dễ dàng**: Tăng/giảm số instance theo traffic, tối ưu chi phí cloud.
3. **Deploy an toàn**: Cập nhật phiên bản mới mà không gây downtime cho người dùng.
4. **Nhất quán**: Dev, Staging, Production dùng chung cấu hình, giảm lỗi "chạy trên máy tôi được".
5. **Tự động hóa**: Kết hợp CI/CD để push code → tự động deploy, giảm thao tác thủ công.

> **Cần đầu tư gì?**

- Thời gian học: **5–7 tuần** cho Phase 1–3 (đủ để vận hành).
- Công cụ: Miễn phí (minikube, kubectl, Helm).
- Cloud: Managed K8s (EKS/GKE/AKS) giá từ ~$70/tháng cho cluster nhỏ (chưa tính worker nodes).
